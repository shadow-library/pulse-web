# Project Directory Structure

## Overview

This document describes the folder structure and conventions for this TanStack Start project.
The guiding principle is **strict separation of concerns by folder purpose** — every folder has
one clear responsibility, and the rule for where to place a file should always have an obvious answer.

## Hard Rules

These are non-negotiable conventions enforced across the entire codebase.

| Rule                                           | Description                                                                                                                                 |
| ---------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| **1 component per file**                       | Each file exports exactly one component. No exceptions.                                                                                     |
| **1 hook per file**                            | Each file contains exactly one hook. No exceptions.                                                                                         |
| **Named exports only**                         | All components and hooks must use named exports. No default exports.                                                                        |
| **`routes/` is routing-only**                  | No components, hooks, or logic live directly in route files — unless the page is simple enough to not warrant a `views/` entry (see below). |
| **`views/` is opt-in**                         | A `views/` entry is only created when needed. A route file may contain inline logic if the page is small.                                   |
| **`*.gen.ts` files are never edited manually** | `api.gen.ts` and `routeTree.gen.ts` are script-generated.                                                                                   |
| **Hook file names are `kebab-case`**           | Applies to `src/hooks/` and all `hooks/` folders inside `views/` and `features/`.                                                           |
| **Promote when shared**                        | If two pages use the same thing, move it to `components/`, `hooks/`, or `features/`.                                                        |

## When to create a `views/` entry for a route

A `views/` folder for a route is **not mandatory**. Create one when any of the following is true:

- The route file is becoming too large to read comfortably
- You need to extract even one component (due to the 1-component-per-file rule)
- You need to extract even one hook (due to the 1-hook-per-file rule)
- Multiple hooks need to be used together in a way that clutters the route file

If none of the above apply, logic and JSX can live directly in the route file.

## Root Structure

```
project-root/
├── public/                         ← Static assets (images, fonts, favicons, etc.)
├── src/                            ← All application source code
├── scripts/                        ← Dev scripts (API generation, linting, setup, etc.)
├── tests/                          ← TBD — Playwright integration tests (see Tests section)
└── ...                             ← Config files (tsconfig, vite, etc.)
```

## `src/` Structure

```
src/
├── routes/                         ← Routing only. No components or logic live here.
├── views/                          ← Page-private code. Created on demand, not upfront.
├── features/                       ← Shared business logic grouped by domain.
├── components/                     ← Shared, domain-agnostic UI components.
├── hooks/                          ← Shared, domain-agnostic hooks.
├── lib/                            ← Utilities, API clients, and configuration.
├── global.css                      ← Global styles.
├── router.tsx                      ← Router setup and configuration.
└── routeTree.gen.ts                ← Auto-generated by TanStack Router. Do not edit.
```

## `src/routes/`

Contains **only** route files. Each route file is responsible for three things only:

1. Defining the route config (`createFileRoute`)
2. Wiring up loaders / search params / route context
3. Composing the page — either inline (small pages) or by importing from `views/`

```
src/routes/
├── __root.tsx                      ← Root layout
├── index.tsx                       ← /
├── users/
│   ├── index.tsx                   ← /users
│   ├── me.tsx                      ← /users/me
│   └── $userId.tsx                 ← /users/:userId
└── dashboard/
    ├── index.tsx                   ← /dashboard
    └── settings.tsx                ← /dashboard/settings
```

### Example — small page (no `views/` entry needed)

```tsx
// routes/users/me.tsx
import { createFileRoute } from '@tanstack/react-router';

function MePage() {
  return <div>Profile</div>;
}

export const Route = createFileRoute('/users/me')({
  component: MePage,
});
```

### Example — larger page (delegates to `views/`)

```tsx
// routes/dashboard/index.tsx
import { createFileRoute } from '@tanstack/react-router';
import { RevenueChart, StatCard, useRevenueData } from '@/views/dashboard/_self';

function DashboardPage() {
  const { data } = useRevenueData();
  return (
    <div>
      <StatCard data={data} />
      <RevenueChart data={data} />
    </div>
  );
}

export const Route = createFileRoute('/dashboard/')({
  component: DashboardPage,
});
```

## `src/views/`

Contains **page-private** code — components, hooks, helpers, constants, and styles used
by exactly one route. Nothing inside `views/` should ever be imported by another view or
by `features/`.

### Naming Convention

The folder structure **mirrors `routes/`** using namespace folders.

- **Namespace folders** (e.g. `users/`) are for grouping only — they never contain files directly.
- **`_self/`** represents the content for the route at that exact segment. Only needed when a route has child routes — otherwise the folder itself is unambiguous.
- When a route gains children, add `_self/` and move existing content into it.

```
src/views/
├── users/
│   ├── _self/                      ← content for /users (has children, so _self is needed)
│   │   ├── components/
│   │   │   └── UserTable.tsx
│   │   ├── hooks/
│   │   │   └── use-users.ts
│   │   ├── users.constants.ts
│   │   ├── users.helpers.ts
│   │   ├── users.module.css
│   │   └── index.ts                ← barrel export
│   ├── me/                         ← content for /users/me (leaf route, no _self needed)
│   │   ├── components/
│   │   │   └── ProfileCard.tsx
│   │   ├── hooks/
│   │   │   └── use-me.ts
│   │   ├── me.constants.ts
│   │   ├── me.helpers.ts
│   │   ├── me.module.css
│   │   └── index.ts
│   └── $userId/                    ← content for /users/:userId (leaf route, no _self needed)
│       ├── components/
│       │   └── UserDetail.tsx
│       ├── hooks/
│       │   └── use-user.ts
│       ├── userId.constants.ts
│       ├── userId.helpers.ts
│       ├── userId.module.css
│       └── index.ts
└── dashboard/
    ├── _self/                      ← content for /dashboard (has children, so _self is needed)
    │   ├── components/
    │   │   ├── RevenueChart.tsx
    │   │   └── StatCard.tsx
    │   ├── hooks/
    │   │   └── use-revenue-data.ts
    │   ├── dashboard.constants.ts
    │   ├── dashboard.helpers.ts
    │   ├── dashboard.module.css
    │   └── index.ts
    └── settings/                   ← content for /dashboard/settings (leaf route)
        ├── components/
        │   └── SettingsForm.tsx
        ├── hooks/
        │   └── use-settings-form.ts
        ├── settings.constants.ts
        ├── settings.helpers.ts
        ├── settings.module.css
        └── index.ts
```

### What lives inside each view folder

| File / Folder      | Purpose                                                                    |
| ------------------ | -------------------------------------------------------------------------- |
| `components/`      | JSX components private to this page. One component per file, named export. |
| `hooks/`           | Hooks private to this page. One hook per file, `kebab-case` filenames.     |
| `<n>.constants.ts` | Constants specific to this page (enums, static config, magic values, etc.) |
| `<n>.helpers.ts`   | Pure helper functions specific to this page                                |
| `<n>.module.css`   | Scoped CSS for this page                                                   |
| `index.ts`         | Barrel export — the only file the route imports from                       |

### When to promote code out of `views/`

| Situation                                                | Move to                  |
| -------------------------------------------------------- | ------------------------ |
| Component is used by 2+ pages                            | `src/components/`        |
| Hook is used by 2+ pages                                 | `src/hooks/`             |
| Logic is shared but domain-specific (e.g. auth, billing) | `src/features/<domain>/` |

## `src/features/`

Contains **shared business logic grouped by domain**. A feature is used by more than one
page and is tied to a specific business capability.

```
src/features/
├── auth/
│   ├── components/
│   │   └── ProtectedRoute.tsx
│   ├── hooks/
│   │   └── use-auth.ts
│   └── index.ts
└── billing/
    ├── components/
    │   └── PlanBadge.tsx
    ├── hooks/
    │   └── use-subscription.ts
    └── index.ts
```

## `src/components/`

Contains **shared, domain-agnostic UI components**. These are purely presentational and
carry no business logic. Think design system components — buttons, modals, form fields, etc.
All components use named exports.

```
src/components/
├── Button/
│   ├── Button.tsx
│   └── Button.module.css
└── Modal/
    ├── Modal.tsx
    └── Modal.module.css
```

## `src/hooks/`

Contains **shared, domain-agnostic hooks** used across multiple features or pages.
File names are `kebab-case`. One hook per file.

```
src/hooks/
├── use-auth-context.ts
├── use-debounce.ts
└── use-pagination.ts
```

## `src/lib/`

Contains utilities, API infrastructure, and configuration.

```
src/lib/
├── api.gen.ts                      ← Auto-generated API functions and types. Do not edit.
└── api-setup.ts                    ← API client setup — pre/post request hooks (auth headers, error handling, etc.)
```

## `scripts/`

Contains developer scripts that are run outside the application.

```
scripts/
├── generate-api.ts                 ← Generates src/lib/api.gen.ts from OpenAPI spec
└── ...
```
